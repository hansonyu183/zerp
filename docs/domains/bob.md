# BOB 领域描述

## 1. 领域定义

**BOB（Business Object Base）** 是 ZERP 的基础业务对象领域，负责维护企业经营活动中可被其他业务领域识别、选择和引用的基础对象。

BOB 是主数据来源，不负责销售、采购、库存、资金收付等交易过程。其他领域只能引用符合本领域生效规则的 BOB 数据。

## 2. 领域目标

- 为各业务领域提供统一、可信的基础业务对象。
- 通过审核控制保证基础数据经过授权后才能用于业务。
- 保留对象的版本和审核轨迹，使历史变化可查询、可追溯。
- 防止未审核或已经失效的数据进入新的业务流程。

## 3. 业务实体

| 实体 | 标识 | 说明 |
| --- | --- | --- |
| 客户 | `customer` | 购买企业产品或服务的个人、企业或组织 |
| 供应商 | `supplier` | 向企业提供产品、服务或其他资源的个人、企业或组织 |
| 员工 | `employee` | 与企业存在任职关系的人员 |
| 产品 | `product` | 可采购、销售、生产或进行库存管理的实物对象 |
| 服务 | `service` | 可采购或销售的非实物业务对象 |
| 资金账户 | `fund-account` | 用于记录现金、银行存款及其他资金的账户 |

后续新增的基础业务实体，如果同样遵循“审核后才可被其他领域引用”的规则，应归入 BOB。

## 4. 核心业务规则

### 4.1 审核后生效

- 新建对象不能直接生效，必须先提交审核。
- 只有审核通过的对象才处于有效状态。
- 草稿、待审核、已驳回和已失效的数据均不能被新的业务引用。
- 提交人与审核人不能是同一人。
- 审核必须记录审核人、审核时间和审核意见。

### 4.2 编辑即失效

- 有效对象一旦发起编辑，当前有效版本立即失效。
- 编辑产生一个新版本，不直接覆盖原版本。
- 新版本必须重新提交审核，审核通过后才成为新的有效版本。
- 新版本审核完成前，该业务对象不存在可供新业务引用的有效版本。
- 新版本被驳回后，原版本不自动恢复有效；需要修正新版本并重新提交审核。

### 4.3 历史数据

- 业务对象和其版本不得物理删除。
- 已经发生的业务记录必须保留所引用的 BOB 对象标识和版本标识。
- 对展示、打印或审计有影响的名称、编码等信息，交易领域应按自身要求保存业务发生时的快照。
- BOB 对象后续编辑或失效不得改变既有业务记录的历史事实。

### 4.4 并发控制

- 同一业务对象同一时间只能存在一个可编辑或待审核的新版本。
- 编辑、提交和审核操作必须校验版本号，禁止用过期数据覆盖新数据。
- 重复提交、重复审核或审核已变化的版本必须返回明确的数据冲突结果。

## 5. 生命周期

### 5.1 状态定义

| 状态 | 标识 | 可编辑 | 可提交审核 | 可被新业务引用 |
| --- | --- | ---: | ---: | ---: |
| 草稿 | `DRAFT` | 是 | 是 | 否 |
| 待审核 | `PENDING` | 否 | 否 | 否 |
| 已驳回 | `REJECTED` | 是 | 是 | 否 |
| 有效 | `EFFECTIVE` | 否 | 否 | 是 |
| 已失效 | `INVALID` | 否 | 否 | 否 |

### 5.2 新建流程

```text
新建 → 草稿 → 提交审核 → 待审核 → 审核通过 → 有效
                                  └→ 审核驳回 → 已驳回
```

### 5.3 编辑流程

```text
有效 → 发起编辑 → 原版本已失效 + 新版本草稿
                                  ↓
                              提交审核
                                  ↓
                               待审核
                         ┌────────┴────────┐
                      审核通过          审核驳回
                         ↓                 ↓
                    新版本有效        新版本已驳回
```

## 6. 对象与版本

BOB 应区分稳定的“业务对象”和随修改产生的“对象版本”。

### 6.1 业务对象

业务对象在整个生命周期中保持唯一，用于表达“这是同一个客户、产品或账户”。建议至少包含：

- `id`：业务对象的稳定唯一标识；
- `entity`：对象所属的实体类型；
- `currentVersionId`：当前版本标识；
- `effectiveVersionId`：当前有效版本标识，无有效版本时为空。

### 6.2 对象版本

每次新建或编辑产生独立版本。建议至少包含：

- `versionId`：版本唯一标识；
- `objectId`：所属业务对象标识；
- `version`：递增版本号；
- `status`：版本状态；
- `data`：该实体的业务字段；
- `createdBy`、`createdAt`：创建审计信息；
- `submittedBy`、`submittedAt`：提交审计信息；
- `reviewedBy`、`reviewedAt`、`reviewComment`：审核信息。

## 7. 领域能力

BOB 中的各业务实体统一提供以下领域能力：

| 能力 | 动作标识 | 说明 |
| --- | --- | --- |
| 查询 | `query` | 按条件分页查询对象或版本 |
| 查看 | `get` | 查看对象详情及指定版本 |
| 新建 | `create` | 创建对象及首个草稿版本 |
| 发起编辑 | `edit` | 使有效版本失效并创建新草稿版本 |
| 保存草稿 | `save` | 保存草稿或已驳回版本 |
| 提交审核 | `submit` | 将可提交版本变为待审核 |
| 审核通过 | `approve` | 将待审核版本变为有效版本 |
| 审核驳回 | `reject` | 将待审核版本变为已驳回版本 |
| 查看版本 | `versions` | 查询对象的全部历史版本 |
| 查看审核记录 | `audit-history` | 查询提交、审核及状态变化记录 |

接口路径遵循项目统一约定：

```text
POST /bob/{entity}/{action}
```

例如：

```text
POST /bob/customer/query
POST /bob/customer/edit
POST /bob/customer/submit
POST /bob/customer/approve
POST /bob/customer/reject
POST /bob/customer/versions
```

## 8. 权限

每个 BOB 实体至少支持以下操作权限：

- 查看；
- 新建；
- 编辑；
- 提交审核；
- 审核通过；
- 审核驳回；
- 查看历史版本和审核记录。

前端权限只控制菜单、按钮和交互可见性，所有状态流转和操作权限必须由后端再次校验。

## 9. 与其他领域的关系

- BOB 向其他领域提供有效基础对象的查询和引用能力。
- 交易领域创建新业务记录时，只能选择 `EFFECTIVE` 版本。
- BOB 不负责决定交易领域如何处理已经引用了失效版本的历史单据。
- 其他领域不得绕过 BOB 的审核状态自行将对象视为有效。
- 领域间使用对象标识和版本标识关联，不依赖可变的名称或编码建立关系。
